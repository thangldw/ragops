# RAGOps v2.0.0 — trustworthy extensible release gates

RAGOps v2.0.0 closes the gap between plugin diagnostics and release decisions,
unifies evaluator behavior across absolute and baseline comparisons, and makes
the optional API fail closed by default.

## Highlights

- Promote selected evaluator metrics into explicit minimum or maximum release
  gates through a version-controlled TOML policy.
- Block findings at a configured severity floor while preserving `critical` as
  the default for existing scenarios.
- Run the same evaluator set and absolute policy on baseline and candidate;
  comparison evidence now includes custom metric deltas.
- Reject non-finite, negative, boolean, and out-of-range numeric input across
  scenarios, responses, traces, policies, and evaluator results.
- Require an API key for protected endpoints unless explicit
  `RAGOPS_INSECURE_DEV_MODE=true` is selected for local development.
- Bound API request bytes and case counts; bind the sample Compose service to
  localhost and require a key.
- Label the Japanese-question suite as a synthetic release-gate fixture rather
  than evidence of semantic Japanese performance.
- Prevent workbench input/API errors from displaying stale PASS/BLOCK evidence;
  accept a tab-only API key without browser persistence.
- Build wheel/source once, publish a reproducible CycloneDX SBOM and checksum
  manifest, attest provenance, then promote the exact verified distributions
  from GitHub Release to PyPI.
- Test Python 3.11, 3.12, and 3.13 and publish complete PyPI discovery metadata.

## Compatibility and migration

CLI and Python users who omit evaluators and evaluation policy retain existing
metric meanings and PASS/BLOCK defaults. Valid scenario, response, trace, and
regression-policy inputs remain accepted.

The API authentication default is intentionally incompatible and makes this a
major release. Before upgrading an API deployment, choose exactly one mode:

- Set a non-empty `RAGOPS_API_KEY` and send it as `x-api-key` on protected
  endpoints; or
- For local, unauthenticated development only, set
  `RAGOPS_INSECURE_DEV_MODE=true` and leave `RAGOPS_API_KEY` unset.

Setting neither mode returns 503 on protected endpoints. Setting both also
returns 503 as a configuration conflict. Health and the static workbench remain
public.

## Known limits

The built-in claim-support evaluator remains lexical rather than semantic
entailment. Provider-backed evaluators still own their calibration,
reproducibility, privacy, and availability guarantees. Input bounds are not a
replacement for reverse-proxy rate limiting, tenant quotas, WAF controls, or a
security assessment. Security scan and penetration testing were explicitly
deferred for this release.
