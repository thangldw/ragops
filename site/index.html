The release problem

A plausible answer is not release evidence.

RAG changes can preserve fluent output while quietly breaking citations, groundedness, latency, cost, or policy behavior.

01

Evidence is scattered

Notebooks, dashboards, prompt logs, and manual reviews rarely share one release contract.

02

Regressions stay hidden

A candidate looks reasonable until it is compared with the last accepted behavior.

03

Decisions lack a reason

A score alone does not explain which gate failed or which case needs attention.

How it works

Turn application output into a release decision.

RAGOps evaluates the system you already have. Retrieval, generation, and orchestration remain outside the dependency-free core.

  1. 01 · RECORDPortable responses or tracesAnswers, citations, retrievals, latency, cost, and metadata stay reviewable.
  2. 02 · EVALUATEVersioned scenario and policyRun deterministic checks, critical findings, and optional external metrics.
  3. 03 · COMPAREAccepted baseline vs candidateMeasure deltas with explicit regression tolerances rather than isolated scores.
  4. 04 · GATEExplainable PASS or BLOCKReturn named reasons and portable evidence for humans and automation.

Open-source core: scenarios, traces, evaluators, comparison, reports, and release gates. Optional adapters: API, browser workbench, providers, and local control-plane alpha.

Known limits

Evidence stays honest about what it proves.

RAGOps reduces release uncertainty. It does not turn synthetic fixtures or lexical checks into production truth.

Lexical is not semantic

Lexical groundedness is overlap, not entailment or human judgment.

Synthetic is not adoption

Fixture results do not prove customer usage, activation, retention, or ROI.

Reference ACL is not identity

The demo role list is not enterprise SSO, RBAC, or a production security boundary.

Local alpha is not hosted SaaS

The control plane is a development surface, not production multi-tenant infrastructure.
