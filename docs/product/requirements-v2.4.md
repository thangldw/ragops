# Requirements v2.4 — adoption experience refresh

## User story

As an engineer evaluating RAGOps, I can understand the product boundary in
seconds, reproduce a PASS/BLOCK decision in five minutes, and follow one clear
path from demo evidence to my own application traces.

## Acceptance criteria

- README and showcase lead with install → evaluate → release decision.
- The stable version, test evidence, and product limitations are consistent.
- The showcase hero is responsive and its primary copy-command interaction works.
- Desktop screenshots come from the rendered implementation, not a static mock.
- Getting-started, walkthrough, portfolio, presentation, video, CV/LinkedIn, and
  launch copy use the same vocabulary and evidence boundary.
- Existing historical ADRs, requirements, acceptance records, and release notes
  remain immutable historical evidence.
- Ruff, full tests, link/asset checks, desktop visual QA, build, clean install,
  and local release evidence pass without GitHub Actions.

## Compatibility and non-goals

This release changes presentation and discoverability only. It does not change
schemas, metrics, release decisions, CLI/API behavior, or open-core boundaries.
It does not invent customer adoption, ROI, semantic-quality, or security claims.
