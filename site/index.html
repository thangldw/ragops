From an ambiguous AI request to a defensible release decision.

01 — Discovery

The request was not “build a chatbot.”

“Use AI to help Japanese field engineers troubleshoot machines faster.”
Trusted evidence

Answers must cite current manuals and escalation policies.

Permission safety

Restricted notes cannot enter retrieval for an unauthorized role.

Controlled agency

External communication requires human approval, not autonomous execution.

Measurable rollout

Quality gates and workflow outcomes determine scale, not demo appeal.

02 — Build

A small system with production-shaped boundaries.

The reference app stays outside the RAGOps core. It owns retrieval and workflow behavior; RAGOps owns evidence, evaluation, comparison, and release policy.

Local-first

Runs without an API key

The offline reference makes CI and review reproducible. Provider generation remains an optional adapter.

Permission-aware

ACL before retrieval

Unauthorized documents are removed before scoring, reducing the chance of downstream leakage.

Approval-aware

Agent does not mean autonomous

Consequential actions return an approval decision. The reference never sends or deletes externally.

03 — Evaluate

Every architecture claim needs a regression test.

The graph-assisted build is the accepted baseline. A lexical-only candidate uses the same questions and release contract, exposing the value of relationship-aware retrieval.

Synthetic evidence validates the harness and architecture comparison. It does not establish semantic model quality, customer adoption, or business ROI.

04 — Deploy responsibly

Recommendation: proceed to customer-reviewed offline pilot. Do not deploy to production.

The integration works and catches a meaningful retrieval regression. Production readiness still requires customer-reviewed data, enterprise identity, shadow-mode observation, incident ownership, and measured workflow impact.
